### [HIGH-01] Oracle Deviation Walk: составное отклонение 50% + 15% = 57.5% — окно для манипулирования ценой

- **Серьёзность:** HIGH
- **Компонент:** `oracle.rs:22-27`, `oracle.rs:36-41`, `constants.rs:42-45`
- **Категория:** Манипулирование оракулом (OWASP A03 — Инъекции / Экономические атаки)
- **Описание:**
  `update_price` использует `MAX_UPDATE_PRICE_DEVIATION_BPS = 5000` (50%), тогда как `mint`/`redeem` используют `MAX_PRICE_DEVIATION_BPS = 1500` (15%). Оба порога проверяют отклонение относительно `last_cached_price`, который является мутабельным. Атакующий может выполнить двухшаговую последовательность в рамках одного блока:
  1. Вызвать `update_price` с ценой Pyth на уровне +50% от кэшированной цены. Кэшированная цена обновляется.
  2. Вызвать `mint` с ценой на уровне +15% от *новой* кэшированной цены, достигая коэффициента 1.50 × 1.15 = 1.725x (72.5% выше оригинала), либо -50%, затем -15% для -57.5%.

  Поскольку `shortSOL_price = k / sol_price`, падение `sol_price` на 57.5% завышает `shortSOL_price` примерно на 135%, позволяя атакующему минтить значительно меньше токенов за USDC (или выкупать по завышенной стоимости).

- **Последствия:** Экономическая манипуляция ценообразованием mint/redeem. Атакующий с выгодным наблюдением оракула Pyth (например, во время высокой волатильности или через специально подготовленное обновление цены) может извлечь средства из хранилища за счёт существующих держателей токенов.

- **Сценарий PoC:**
  1. Кэшированная цена = $100. Реальная цена = $100.
  2. Атакующий вызывает `update_price` с ценой Pyth = $50 (в пределах 50% отклонения). Кэш обновляется до $50.
  3. Атакующий вызывает `mint` с ценой Pyth = $42.50 (в пределах 15% от $50). `shortSOL_price` рассчитывается с $42.50 в знаменателе вместо $100.
  4. `shortSOL_price = k * 1e9 / 42.5e9` — примерно в 2.35x выше, чем при $100. Атакующий получает в 2.35x меньше токенов за тот же USDC, но *стоимость* при возврате цены составит 2.35x.
  5. Когда цена возвращается, атакующий выкупает по истинной цене с прибылью.

- **Рекомендации по исправлению:**
  Реализовать абсолютную якорную цену (например, TWAP или неизменяемую «последнюю известную хорошую» цену, которая может двигаться только в узком диапазоне за единицу времени). Альтернативно — добавить задержку между `update_price` и `mint`/`redeem`, либо использовать единый порог отклонения для всех операций:

  ```rust
  // Option A: Single deviation for all operations
  pub const MAX_PRICE_DEVIATION_BPS: u64 = 1500; // 15% everywhere

  // Option B: Time-weighted cooldown
  pub const UPDATE_PRICE_COOLDOWN_SECS: i64 = 60;
  // In mint/redeem:
  require!(
      clock.unix_timestamp - pool.last_price_update_timestamp >= UPDATE_PRICE_COOLDOWN_SECS,
      SolshortError::PriceUpdateTooRecent
  );
  ```

- **Статус:** Открыт

### [HIGH-02] Инфляция долей первого LP-вкладчика (без dead shares)

- **Серьёзность:** HIGH
- **Компонент:** `fees.rs:104-119` (`calc_lp_shares`), `add_liquidity.rs:101`
- **Категория:** Экономическая атака (Небезопасный дизайн — A04)
- **Описание:**
  Когда `lp_total_supply == 0`, доли минтятся в соотношении 1:1 с `usdc_amount`. Первый вкладчик может:
  1. Внести минимум ($100 USDC), получить 100_000_000 LP-долей.
  2. Перевести крупную сумму USDC напрямую в хранилище (не через `add_liquidity`), раздувая `vault_balance` относительно `lp_principal`.

  Однако расчёт долей использует `lp_principal` (а не `vault_balance`) в качестве знаменателя: `shares = usdc_amount * lp_total_supply / lp_principal`. Поскольку прямой перевод НЕ увеличивает `lp_principal`, вектор donation-атаки через математику на основе `lp_principal` существенно снижен.

  **Остаточный риск:** Если протокол когда-либо перейдёт на использование `vault_balance` для ценообразования долей (распространённый паттерн), или если распределение комиссий фондирования непропорционально раздует `fee_per_share_accumulated` до появления второго LP-вкладчика, возникнет вектор инфляции.

  `MIN_LP_DEPOSIT = $100` обеспечивает дополнительную защиту, делая атаку экономически невыгодной.

- **Последствия:** Опытный первый вкладчик может захватить непропорционально большую долю будущего распределения комиссий. При текущей математике на основе `lp_principal` прямое пополнение хранилища не раздувает цену доли, однако защита недостаточно надёжна (отсутствует паттерн dead shares).

- **Сценарий PoC:**
  1. Атакующий — первый LP. Вносит минимум $100.
  2. После накопления комиссий растёт `fee_per_share_accumulated`.
  3. Второй LP вносит $10,000. Доли = `10_000e6 * 100e6 / 100e6 = 10_000e6`.
  4. Если атакующий каким-либо образом раздул `lp_principal`, второй вкладчик получит меньше долей. Текущий код устойчив, но не обладает защитой в глубину.

- **Рекомендации по исправлению:**
  Реализовать паттерн dead shares (виртуальное смещение), используемый в ERC-4626:

  ```rust
  pub fn calc_lp_shares(usdc_amount: u64, lp_total_supply: u64, lp_principal: u64) -> Result<u64> {
      const VIRTUAL_SHARES: u64 = 1_000; // 1e3 dead shares
      const VIRTUAL_ASSETS: u64 = 1_000; // 1e3 dead USDC (0.001 USDC)

      let total_shares = lp_total_supply.checked_add(VIRTUAL_SHARES).ok_or(/*...*/)?;
      let total_assets = lp_principal.checked_add(VIRTUAL_ASSETS).ok_or(/*...*/)?;

      (usdc_amount as u128)
          .checked_mul(total_shares as u128).ok_or(/*...*/)?
          .checked_div(total_assets as u128).ok_or(/*...*/)?
          .try_into().map_err(|_| /*...*/)
  }
  ```

- **Статус:** Открыт (рекомендована защита в глубину)

### [MEDIUM-01] `migrate_pool` — жёстко заданные байтовые смещения и хрупкая привязка к сериализации

- **Серьёзность:** MEDIUM
- **Компонент:** `migrate_pool.rs:83-97`
- **Категория:** Ошибки конфигурации безопасности (A05)
- **Описание:**
  Обработчик `migrate_pool` использует жёстко заданные байтовые смещения для нахождения поля `min_lp_deposit` по смещению `8 + 205 + 64 = 277`. Эти смещения тесно связаны с Borsh-раскладкой сериализации `PoolState`. Если какое-либо поле будет переупорядочено, изменён его размер или новое поле вставлено перед LP-полями, смещение станет некорректным и `min_lp_deposit` будет записан в неверную область памяти, что потенциально повредит другие поля состояния.

  Комментарий `// lp_mint offset: 8+32+16+2+8+8+8+8+8+32+32+1+8+8+1+1+32 = 213` даёт значение 213, но код использует `8 + 205 + 64 = 277` для `min_deposit_offset`, подразумевая, что старые поля занимают 205 байт. Расхождение между комментарием (213) и кодом (205) указывает на возможную ошибку расчёта в прошлом, хотя текущий код, по-видимому, обращается к нужному полю.

- **Последствия:** Если `PoolState` когда-либо будет изменён, `migrate_pool` может бесшумно повредить состояние пула, потенциально обнулив критически важные для безопасности поля, такие как `fee_bps` или `authority`.

- **Сценарий PoC:**
  1. Разработчик добавляет новое поле `u64` перед `pending_authority` в `PoolState`.
  2. `migrate_pool` по-прежнему записывает `min_lp_deposit` по смещению 277.
  3. Запись повреждает поле `lp_total_supply` или `fee_per_share_accumulated`.
  4. LP-система становится неплатёжеспособной.

- **Рекомендации по исправлению:**
  Заменить жёстко заданные смещения на `try_deserialize` / `try_serialize` из Anchor, либо вычислять смещения с помощью `core::mem::offset_of!` (Rust 1.77+):

  ```rust
  // Preferred: deserialize, modify, serialize
  let mut pool: PoolState = PoolState::try_deserialize(&mut &data[..])?;
  pool.min_lp_deposit = MIN_LP_DEPOSIT;
  pool.try_serialize(&mut &mut data[..])?;
  ```

  Если миграция — одноразовая операция, уже выполненная для всех существующих пулов, рекомендуется полностью удалить инструкцию для сокращения поверхности атаки.

- **Статус:** Открыт (рекомендуется удаление, если миграция завершена)

### [MEDIUM-02] `funding_config` — опциональный в `mint`/`redeem`, обход устаревшего k

- **Серьёзность:** MEDIUM
- **Компонент:** `mint.rs:68-73`, `redeem.rs:59-64`
- **Категория:** Небезопасный дизайн (A04)
- **Описание:**
  В обоих account-структурах `MintShortSol` и `RedeemShortSol` `funding_config` объявлен как `Option<Account<'info, FundingConfig>>`. Если клиент не передаёт аккаунт `FundingConfig`, встроенное начисление фондирования полностью пропускается, и `k` остаётся на своём последнем обновлённом значении.

  Это означает, что пользователь может намеренно не передать аккаунт `FundingConfig`, чтобы минтить/выкупать по устаревшему (более высокому) значению `k`, фактически избегая затухания funding rate, которое должно непрерывно снижать `k`.

- **Последствия:** Пользователи могут избежать штрафов funding rate, не передавая опциональный аккаунт. Со временем, если keeper работает нерегулярно и пользователи постоянно опускают `funding_config`, фактическое значение `k`, используемое для ценообразования, расходится с предполагаемым затухающим по времени `k`. Это создаёт нечестное преимущество для осведомлённых пользователей перед LP, которые ожидают затухания `k`.

- **Сценарий PoC:**
  1. `k` последний раз начислялся 12 часов назад. При ставке 10 bps/день `k` должен был затухнуть примерно на 0.05%.
  2. Пользователь вызывает `mint` БЕЗ передачи `funding_config`.
  3. `apply_funding_inline` не вызывается. Минт использует устаревшее (более высокое) `k`.
  4. Более высокое `k` означает более высокую `shortSOL_price`, поэтому пользователь получает меньше токенов за тот же USDC (невыгодно для mint).
  5. И наоборот, для `redeem`: устаревшее более высокое `k` означает более высокую `shortSOL_price`, больше USDC за токен — выгодно для выкупающего за счёт LP.

- **Рекомендации по исправлению:**
  Сделать `funding_config` обязательным, если он существует (проверить on-chain, инициализирован ли PDA):

  ```rust
  // After applying optional funding, verify k freshness
  if ctx.accounts.funding_config.is_none() {
      // Check if FundingConfig PDA exists on-chain
      let (funding_pda, _) = Pubkey::find_program_address(
          &[FUNDING_SEED, pool.key().as_ref()],
          ctx.program_id,
      );
      // If it exists, require it to be passed
      // (This requires remaining_accounts check or a pool-level flag)
  }

  // Simpler: add a flag to PoolState
  // pub funding_enabled: bool,
  // require!(!pool.funding_enabled || ctx.accounts.funding_config.is_some(),
  //     SolshortError::FundingConfigRequired);
  ```

- **Статус:** Открыт

### [MEDIUM-03] `accrue_funding` — `factor_num` может приблизиться к нулю через `saturating_sub`

- **Серьёзность:** MEDIUM
- **Компонент:** `accrue_funding.rs:26-29`
- **Категория:** Целочисленная арифметика (A03 — Инъекции/Логика)
- **Описание:**
  Формула затухания фондирования вычисляет:
  ```
  denom = SECS_PER_DAY * BPS_DENOMINATOR = 864_000_000
  reduction = rate_bps * elapsed_to_apply
  factor_num = denom.saturating_sub(reduction)
  ```

  При `MAX_FUNDING_RATE_BPS = 100` и `MAX_FUNDING_ELAPSED_SECS = 30 дней = 2_592_000`:
  ```
  reduction = 100 * 2_592_000 = 259_200_000
  factor_num = 864_000_000 - 259_200_000 = 604_800_000  (ok, ~70%)
  ```

  Однако проверка `require!(factor_num > 0)` на строке 29 защищает от нуля, но `saturating_sub` маскирует истинную арифметику. Если параметры когда-либо изменятся (например, повышение `MAX_FUNDING_RATE_BPS` или увеличение `MAX_FUNDING_ELAPSED_SECS`), `reduction` может превысить `denom`, и `saturating_sub` выдаст 0, вызвав срабатывание require. Проблема в том, что `saturating_sub` скрывает переполнение вместо явного отказа.

  Кроме того, при текущих параметрах: если keeper неактивен 30 дней при ставке 100 bps/день, `k` затухает до `k * 604_800_000 / 864_000_000 = k * 0.7`. За несколько 30-дневных циклов это составно стремится к нулю.

- **Последствия:** Если `factor_num` насыщается до очень малого значения, `k` может затухнуть почти до нуля за несколько циклов, делая `shortSOL_price` фактически бесконечной и нарушая экономику протокола.

- **Сценарий PoC:**
  1. Ставка = 100 bps/день, keeper неактивен 30 дней.
  2. `k_new = k * 0.70` за цикл.
  3. После 10 циклов (300 дней без keeper): `k_new = k * 0.70^10 = k * 0.028`.
  4. `shortSOL_price` раздувается в 35x. Выкупы истощают хранилище.

- **Рекомендации по исправлению:**
  Добавить минимальный порог `k` и использовать `checked_sub` вместо `saturating_sub`:

  ```rust
  pub const MIN_K: u128 = 1_000_000; // Minimum k value

  let reduction: u128 = cfg.rate_bps as u128 * elapsed_to_apply as u128;
  let factor_num = denom.checked_sub(reduction)
      .ok_or(error!(SolshortError::MathOverflow))?;

  let new_k = pool.k
      .checked_mul(factor_num).ok_or(/*...*/)?
      .checked_div(denom).ok_or(/*...*/)?;

  pool.k = new_k.max(MIN_K);
  ```

- **Статус:** Открыт

### [MEDIUM-04] `claim_lp_fees` использует `saturating_sub` для `total_lp_fees_pending`

- **Серьёзность:** MEDIUM
- **Компонент:** `claim_lp_fees.rs:88`
- **Категория:** Нарушение бухгалтерского инварианта (A04)
- **Описание:**
  Строка 88: `pool.total_lp_fees_pending = pool.total_lp_fees_pending.saturating_sub(amount);`

  Если `amount > total_lp_fees_pending` из-за накопленных ошибок округления в аккумуляторе комиссий, `saturating_sub` бесшумно ограничивает значение нулём вместо отмены операции. Это маскирует нарушение бухгалтерского инварианта: `total_lp_fees_pending` всегда должно быть >= суммы всех индивидуальных `pending_fees`.

  Инструкция `withdraw_fees` использует `total_lp_fees_pending` как защиту резерва:
  ```rust
  let lp_reserved = pool.lp_principal.checked_add(pool.total_lp_fees_pending)?;
  ```
  Если `total_lp_fees_pending` ошибочно равен нулю из-за насыщающего недополнения, администратор может вывести USDC, которые должны были быть зарезервированы для выплат LP-комиссий.

- **Последствия:** Дрейф бухгалтерии со временем. В худшем случае LP-провайдеры не могут получить полную сумму причитающихся им комиссий, потому что администратор вывел средства, которые должны были быть зарезервированы.

- **Сценарий PoC:**
  1. Аккумулятор комиссий распределяет 1000 USDC между 10 LP.
  2. Из-за потери точности при делении на `SHARE_PRECISION` `sum(individual pending_fees)` = 1001 USDC.
  3. Первые 9 LP получают свои комиссии. `total_lp_fees_pending` уменьшается нормально.
  4. 10-й LP запрашивает 101 USDC. `saturating_sub` выдаёт 0 вместо отмены при недополнении.
  5. `total_lp_fees_pending = 0`. `withdraw_fees` администратора больше не резервирует средства для будущих LP-комиссий.

- **Рекомендации по исправлению:**
  Использовать `checked_sub` и обработать граничный случай явно:

  ```rust
  pool.total_lp_fees_pending = pool
      .total_lp_fees_pending
      .checked_sub(amount)
      .unwrap_or_else(|| {
          msg!("WARN: total_lp_fees_pending underflow by {}", 
               amount.saturating_sub(pool.total_lp_fees_pending));
          0
      });
  ```

### [LOW-01] Потеря точности `fee_per_share_accumulated` при малых комиссиях

- **Серьёзность:** LOW
- **Компонент:** `fees.rs:123-141` (`accumulate_fee`)
- **Категория:** Арифметическая точность (A04)
- **Описание:**
  Дельта аккумулятора комиссий вычисляется как:
  ```
  delta = fee_amount * SHARE_PRECISION / lp_total_supply
  ```
  При `SHARE_PRECISION = 1e12`, если `fee_amount = 1` (1 микро-USDC) и `lp_total_supply = 1e12` (1М LP-токенов при 1e6 десятичных), `delta = 1 * 1e12 / 1e12 = 1`. Это точный результат.

  Однако, если `lp_total_supply > 1e12`, `delta` округляется до 0, и комиссия фактически теряется (остаётся в хранилище, но никогда не может быть получена LP). При `lp_total_supply = 1e13` (10М LP-токенов) любая комиссия менее 10 микро-USDC ($0.00001) теряется.

- **Последствия:** Пренебрежимо мало при текущем масштабе. Становится актуальным только при очень большом LP-supply с очень малыми отдельными комиссионными событиями (например, комиссии в 1 микро-USDC на LP-пуле в $10М).

- **Рекомендации по исправлению:**
  Допустимо при текущем масштабе. Для защиты в глубину — накапливать пылевые комиссии в отдельном счётчике и распределять при достижении порога:

  ```rust
  let delta = (fee_amount as u128)
      .checked_mul(SHARE_PRECISION)?
      .checked_div(pool.lp_total_supply as u128)?;
  if delta == 0 {
      pool.dust_fees = pool.dust_fees.checked_add(fee_amount)?;
      return Ok(());
  }
  ```

- **Статус:** Принято к сведению (допустимый риск)

### [LOW-03] `update_price` без ограничения доступа — вектор MEV-гриферства

- **Серьёзность:** LOW
- **Компонент:** `update_price.rs:24`, `update_price.rs:26-49`
- **Категория:** MEV / Гриферство (A01 — Управление доступом)
- **Описание:**
  `update_price` требует только обычного `Signer` (любой кошелёк), а не authority пула. Широкое окно отклонения в 50% означает, что любой может сдвинуть кэшированную цену к крайнему значению в пределах этого диапазона. Хотя `last_oracle_timestamp` НЕ обновляется (предотвращая обход ограничения частоты), изменение кэшированной цены сохраняется и влияет на последующие проверки отклонения в `mint`/`redeem`.

  MEV-бот может вызвать `update_price` непосредственно перед транзакцией `mint` пользователя, чтобы сдвинуть кэшированную цену, а затем «обернуть» транзакцию пользователя (sandwich-атака).

- **Последствия:** Гриферство: атакующий может сместить кэшированную цену к границе 50%-ного диапазона, вызывая отказ легитимных транзакций `mint`/`redeem` с ошибкой `PriceDeviationTooHigh`, если цена Pyth сместилась. В сочетании с HIGH-01 это позволяет двухшаговый сдвиг цены.

- **Рекомендации по исправлению:**
  Либо ограничить `update_price` только authority, либо сузить отклонение до уровня `mint`/`redeem` (15%):

  ```rust
  // Option A: Authority-only
  #[account(has_one = authority)]
  pub pool_state: Account<'info, PoolState>,
  pub authority: Signer<'info>,

  // Option B: Same deviation as mint/redeem
  let oracle_price = oracle::get_validated_price(
      &ctx.accounts.pyth_price,
      pool.last_oracle_price,
  )?;
  ```

- **Статус:** Открыт

### [LOW-04] `realloc(target_len, false)` — неинициализированная память в мигрированном диапазоне

- **Серьёзность:** LOW
- **Компонент:** `migrate_pool.rs:59`
- **Категория:** Безопасность памяти (A05)
- **Описание:**
  `pool_info.realloc(target_len, false)` НЕ заполняет новые байты нулями. Далее код вручную обнуляет `data[lp_start..target_len]` на строке 87, что корректно. Однако существует тонкое окно между `realloc` (строка 59) и циклом обнуления (строка 87), в котором перевод ренты (строки 62-76) может завершиться ошибкой, оставив аккаунт в частично мигрированном состоянии с неинициализированной памятью.

  Если перевод ренты завершается ошибкой, функция возвращает ошибку, и realloc откатывается средой исполнения Solana (атомарность на уровне транзакции). Таким образом, на практике это безопасно.

- **Последствия:** Отсутствуют на практике благодаря атомарности транзакций. Параметр `false` является плохой практикой кода, которая может быть опасна при копировании в другое место.

- **Рекомендации по исправлению:**
  Использовать `realloc(target_len, true)` для безопасности, принимая незначительные затраты CU:

  ```rust
  pool_info.realloc(target_len, true)?; // zero-init new bytes
  ```

- **Статус:** Принято к сведению (безопасно благодаря атомарности транзакций, но рекомендуется `true` для ясности)

### [INFO-02] Ограничение частоты на основе временной метки оракула, а не слота

- **Серьёзность:** INFORMATIONAL
- **Компонент:** `mint.rs:84-90`, `redeem.rs:80-86`
- **Категория:** Допущения о временны́х метках
- **Описание:**
  Ограничение частоты в 2 секунды (`MIN_ACTION_INTERVAL_SECS`) сравнивает `clock.unix_timestamp` с `pool.last_oracle_timestamp`. В Solana `unix_timestamp` имеет гранулярность на уровне слота (~400мс). Две транзакции в соседних слотах могут иметь временные метки с разницей менее 2 секунд, обеспечивая эффективное ограничение частоты. Однако в пределах одного слота `unix_timestamp` идентичен, поэтому проверка `clock.unix_timestamp - last_oracle_timestamp >= 2` всегда будет отклонена для транзакций в том же слоте после первой.

  Это фактически *более строгая* гарантия, чем предполагалось — по сути один mint/redeem за слот после первого.

- **Последствия:** Отсутствуют. Ограничение частоты работает как предполагалось или лучше.

- **Статус:** Принято к сведению
